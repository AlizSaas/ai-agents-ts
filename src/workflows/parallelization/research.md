=== Research on AI coding agents ===

=== Technical Architecture ===
AI coding agents are LLMs wrapped in a *scaffold* (controller code) that runs an iterative “sense → plan → act → verify” loop: the orchestrator builds a task state (user goal, repo snapshot, constraints, prior actions), injects *context* via retrieval (embeddings over the codebase/docs/issues) plus lightweight “memory” (summaries, to-do list, failed attempts), then prompts the model to either (a) propose a code change or (b) emit a structured *tool call* (JSON/function calling) to do concrete work—read/search files, run ripgrep, compile/tests, query package docs, open a PR, etc. ([help.openai.com](https://help.openai.com/en/articles/8555517-function-calling-in-the-openai-api%23.eps?utm_source=openai)) Many modern agents reduce brittle multi-tool chatter by giving the model a controlled *execution harness/sandbox* (often containerized) where it can run commands and tests safely, capture stdout/stderr/artifacts, checkpoint/rehydrate state after failures, and feed results back to the model for the next step. ([openai.com](https://openai.com/index/the-next-evolution-of-the-agents-sdk/?utm_source=openai)) Under the hood, the “intelligence” is mostly in the loop design: common primitives include ReAct-style interleaving of reasoning and actions, generate–test–repair cycles that iterate until tests pass, plan–execute decompositions, and retry/tree-search variants; the agent selects actions based on tool outputs, updates its state/memory, and eventually produces a patch (diff) plus an explanation grounded in test logs and retrieved code context. ([arxiv.org](https://arxiv.org/abs/2604.03515?utm_source=openai))

=== Business Use Cases ===
AI coding agents are now used in businesses mainly to accelerate “software delivery throughput” across the SDLC: engineers use in-IDE agents (e.g., GitHub Copilot, Cursor) for code completion, refactors, translating between languages/frameworks, writing unit/integration tests, generating documentation, and quickly exploring unfamiliar codebases; platform/DevOps teams use them to produce infrastructure-as-code, CI/CD configs, scripts, and incident runbooks; and more “autonomous” agents (e.g., Devin) are piloted for multi-step work like migrations, modernization, repetitive backlog chores, and semi-automated incident response in sandboxed environments with human review. Evidence of who’s using them includes: Microsoft has said nearly all of its own engineers use GitHub Copilot, and GitHub reports broad enterprise uptake while Copilot is positioned as an enterprise developer tool integrated into GitHub workflows (coding plus PR assistance). ([azure.microsoft.com](https://azure.microsoft.com/en-us/blog/github-universe-2025-where-developer-innovation-took-center-stage/?utm_source=openai)) Cursor markets enterprise adoption (naming Salesforce, NVIDIA, and PwC) for faster shipping with context-aware repo-level assistance. ([cursor.com](https://cursor.com/blog/enterprise?utm_source=openai)) Atlassian’s “Rovo Dev” positions an agent for planning, coding, and reviews inside Jira/Atlassian context (i.e., using enterprise work-item data to drive engineering execution). ([atlassian.com](https://www.atlassian.com/solutions/devops/ai-innovation?utm_source=openai)) Devin’s vendor materials highlight use on complex, multi-step engineering tasks like code migrations/modernization and operational workflows. ([devin.ai](https://devin.ai/?utm_source=openai)) Academic/industry evaluations also document enterprise deployments and measured impacts (e.g., ZoomInfo’s evaluation of Copilot in production). ([arxiv.org](https://arxiv.org/abs/2501.13282?utm_source=openai))

=== Risks & Criticism ===
AI coding agents can fail in ways that are easy to miss because their output often *looks* plausible: they may hallucinate APIs, edge-case logic, or even dependencies (including typosquatted packages), creating reliability and software-supply-chain risk; they can also reproduce insecure patterns learned from training data (e.g., weak input validation, unsafe deserialization, insecure auth/session handling), and multiple empirical studies have found AI-generated code frequently contains security weaknesses in real projects. ([arxiv.org](https://arxiv.org/abs/2310.02059?utm_source=openai)) Agents that have tool access (IDE plugins, shell, file system, package managers, CI) expand the attack surface: prompt-injection or malicious content in a repo/webpage/ticket can steer the agent to leak secrets, change configs, or take unsafe actions, and security researchers have reported concrete IDE/agent vulnerability classes enabling data exfiltration and even remote code execution. ([owasp.org](https://owasp.org/www-community/attacks/PromptInjection?utm_source=openai)) There are also governance and legal criticisms: generated code can be hard to audit and attribute, may encourage over-trusting “confident” wrong answers, and can raise licensing/IP concerns if it closely matches copyrighted or restrictive-licensed training examples. ([techcrunch.com](https://techcrunch.com/2022/12/28/code-generating-ai-can-introduce-security-vulnerabilities-study-finds/?utm_source=openai))
